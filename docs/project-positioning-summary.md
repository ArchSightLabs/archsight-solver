# ArchSight Solver 项目定位与开源价值总结

## 一句话结论

ArchSight Solver 不是一个“梁计算器”，而是一个面向结构工程师、教师和进阶学习者的开源结构力学求解器工作台。

它的核心价值，是把结构计算从黑盒、碎片化、难复核的状态，拉回到**可验证、可复现、可复用、可交付**的工程路径上。

## 项目现在做的是什么

当前仓库已经形成了比较完整的工作台形态，不只是单点求值。

- 后端提供统一计算入口，覆盖梁系、二维平面框架、二维平面桁架、预览和导出。
- 前端不是简单表单，而是带有工作区、模块切换、结果面板、模板库的工程工作台。
- 项目已经把“计算 -> 看图 -> 保存模板 -> 导出”串成闭环。
- 规范层已经明确：梁系、二维平面框架、二维平面桁架是三个核心模块。

这意味着它已经不是“演示一个公式”，而是在做一个能被反复使用的行业工具底座。

## 核心功能

### 1. 梁系求解

支持连续梁、简支梁、悬臂梁等基础梁系分析，覆盖均布荷载、集中荷载、线性分布荷载等常见工况。

这部分是项目最稳的核心，因为它直接决定工具是否可信。

### 2. 二维平面框架

当前已经进入框架分析能力，支持门式刚架模板和自定义二维框架建模。

这一步很重要，因为它把项目从“梁工具”推进到了“平面杆系工具”，行业适用面会明显变大。

### 3. 二维平面桁架

项目已经进入二维平面桁架分析能力，适合典型屋架、支撑体系和教学算例的快速复核。

这一步的意义不是扩大版图，而是把“典型二维结构”补齐成三个互相支撑的核心模块。

### 4. 结果表达

项目不只算数值，还输出结构图、荷载图、内力图、挠度曲线、节点位移和工程摘要。

这比单纯给一个结果值更有用。工程人员要的是能复核、能解释、能交付。

### 5. 模板库与工作流复用

模板库能保存、恢复、复制、设为基准、删除。

这让它更像一个真实工作台，而不是一次性的在线计算器。常用工况能沉淀下来，后续复算速度会快很多。

### 6. 计算书导出

支持导出 WORD 计算书，并且应持续强化输入参数、图形、验算过程和验证结果的完整性。

对专业用户来说，这一点很关键。工程软件如果不能导出可交付文档，基本就停留在半成品。

## 项目的开源价值

我判断是：**高于一般工具项目，属于结构工程细分领域里值得长期维护的开源基础设施**。

它的开源价值主要在这几方面：

- **透明**：结果链路公开，用户知道怎么来的。
- **可复现**：同样输入可以重复得到同样输出。
- **可验证**：可以用回归算例和基准案例持续校验。
- **可扩展**：可以在三类核心模块之内继续完善算例、入口、导出和交互。
- **可复用**：模板、快照、导出、API 都能复用到具体工作流里。

如果这个项目继续做稳，它会成为一种“结构工程公共底座”，而不是仅供展示的 demo。

## 对行业有没有帮助

有，而且帮助点很明确。

### 对结构工程师

- 减少重复录参。
- 方便快速复核常见工况。
- 让结果更容易解释给同事、甲方、审查方。

### 对教师和学生

- 更容易把结构力学的抽象问题变成可视化结果。
- 方便做算例教学、作业验证、课程演示。

### 对中小团队

- 可以把常见项目模板化。
- 可以把计算过程标准化，降低人员差异带来的风险。

### 对行业生态

- 提供一个开源、可审计、可贡献的参考实现。
- 让一些本来只能依赖封闭工具的场景，出现替代选项。

## 这个项目做好之后的作用

如果继续做对，它的作用不是“再造一个大软件”，而是做出一个更现代的行业计算底座。

具体来说，它可以成为：

1. 结构工程的快速验证工具。
2. 教学和培训的可视化工具。
3. 团队内部的标准工况计算平台。
4. 公开的参考实现，用来推动行业对“可验证计算”的认知。

这个方向的意义很实在。它不是为了抢所有市场，而是先在一个明确的问题上做到强。

## 项目最适合的定位

我建议你把它长期定位成：

> **Web 原生、透明可验证的开源结构力学求解器工作台**

这个定位有几个好处：

- 足够专业。
- 足够清晰。
- 足够开放。
- 足够适合围绕三个核心模块持续打磨。
- 不会一开始就被“大而全商业软件”拖死。

## 开源范围与外部源码边界

项目不应继续按“所有能力都进入当前主仓库”的方式扩张。更稳的路线是：

- 当前主仓库：梁系、二维平面框架、二维平面桁架、基础计算书、公开 benchmark、错误契约。
- 后续单独立项：私有部署、品牌化计算书、AI 建模助手、规范校核、批量工程和未来三维求解。
- External Source Boundary：公开仓库不包含第三方求解器源码、git submodule 或 `external/` 参考源码树。

这样既保留开源带来的可信度和传播能力，也避免把尚未产品化的扩展能力无边界并入主仓库。

## 适合的演进方向

建议的演进顺序是：

1. 把梁系、二维平面框架、二维平面桁架作为三个核心模块同步做强。
2. 为每个模块补齐高频工况的便捷入口和默认参数。
3. 把 WORD 计算书和结果页交互做到真正可复核、可交付。
4. 用基准算例和真实案例闭环持续收紧可信度。

这个顺序比“先做很多模块”更靠谱。先把底层可信度和交付完整度做出来，后面的优化才有意义。

## 更稳的阶段规划

下面这版规划不再按“功能清单越排越长”的方式展开，而是先锁边界，再打磨主线，再做真实验证。

### Stage 0，叙事收口与遗留归档

- 锁定主定位为“Web 原生、透明可验证的开源结构力学求解器工作台”
- 锁定对外主线为梁系、二维平面框架、二维平面桁架三个核心模块
- 把敏感性分析和模板库明确降为辅助能力，不和三大主线抢优先级
- 取消登录/JWT 探索项，不再作为当前产品能力；运行时认证服务、临时规格和 `PyJWT` 依赖已清理，未来统一身份能力作为独立集成议题处理
- 清理掉会稀释定位的内容，比如影响线、动力、稳定、插件、外部引擎、规范设计、施工安全、云协作、多用户

### Stage 1，核心模块收口

- 把连续梁、简支梁、悬臂梁与典型框架、典型桁架的输入、求解、输出链路做稳
- 为三个核心模块补齐高频工况入口、预置模型和默认参数
- 统一单位转换、参数校验、错误返回和结果摘要字段
- 把典型工况沉淀成可复现算例，不停留在“能跑”

### Stage 2，结果表达与交付闭环

- 让结果图、内力图、挠度曲线、摘要和校核结论在一个页面里能看懂
- 让用户知道“为什么是这个结果”，而不是只给一个数
- 把 WORD 计算书打磨成真正可复核、可交付的输出
- 把模板库和快照能力变成常用工作流，但仍保持辅助能力定位

### Stage 3，回归门禁与开源硬化

- 建立更完整的 benchmark cases，覆盖典型梁系、框架和桁架算例
- 把回归测试、错误契约、lint、类型检查和文档一致性变成门禁
- 补齐安装、开发、测试、算例说明和贡献指南
- 让仓库能被一个陌生开发者快速跑起来

### Stage 4，真实试点与下一轮取舍

- 找 3 到 5 位真实用户试用，最好覆盖工程和教学两类
- 收集反馈，重点看哪里不好用、哪里不可信、哪里不够快
- 只有在主线稳定后，才决定是否提升某个辅助能力
- 如果核心链路仍存在明显短板，继续补齐，不引入新的分析域

## 当前明确不做的方向

为了保持目标收敛，当前仓库不把以下内容作为项目目标：

- 影响线
- 动力分析
- 稳定分析
- 插件化生态
- OpenSeesPy 或其他外部引擎接入
- 将 Frame3D / 三维杆系纳入当前开源核心
- 规范设计、施工安全业务
- 云协作、多用户与权限系统

这些方向一旦进入主路线，就会明显稀释当前项目的核心价值。
