# 支座类型与图例

> 本文说明 ArchSight Solver 中梁系、平面桁架、平面框架的支座类型、自由度约束和图例口径。它面向开源用户、贡献者、产品和研发，不替代结构力学教材、结构设计规范或具备资质人员的工程复核。

## 结论

梁系、平面桁架和平面框架没有使用完全相同的一套支座能力，这是合理的，不是天然问题。

结构分析里的“支座类型”本质上不是图标，而是对节点或梁端自由度的约束。不同分析对象拥有的自由度不同，所以同一个名字在不同对象里会对应不同的约束集合：

| 分析对象 | 当前自由度 | 支座能力边界 |
|---|---|---|
| 梁系 | `v / θz` | 只处理竖向位移和截面转角，不处理水平位移 |
| 平面桁架 | `ux / uy` | 只处理节点平动，不处理转角和弯矩边界 |
| 平面框架 | `ux / uy / rz` | 可约束水平位移、竖向位移和平面转角，也支持部分弹性约束 |

业界通常也是这样处理：支座名称是工程简称，真正进入求解器的是自由度约束矩阵。梁单元、框架单元、桁架单元自由度不同，支座类型自然不能强行完全一致。

当前系统更准确的判断是：

- **不是错误**：三类对象的支座自由度不同，符合分析模型假定。
- **是说明风险**：如果界面、图例、报告只显示“铰支座/滚动支座/固结支座”，不显示约束自由度，用户会误以为同名支座在三类对象中完全等价。
- **需要规范化**：应保留共享支座目录，但所有图例、报告和编辑器都应按“支座名称 + 约束自由度 + 适用对象”展示。

## 支座、支承、支撑的区别

中文语境里这几个词容易混用，但在产品和文档中最好区分：

| 词 | 建议含义 | 产品中怎么用 |
|---|---|---|
| 支座 | 结构与外部边界的连接，提供位移或转角约束 | 用于 `supportType`、支座反力、边界条件 |
| 支承 | 比“支座”更泛的承托或边界概念 | 可在说明文字中使用，不建议作为枚举名 |
| 支撑 | 常指结构内部的支撑杆、斜撑、抗侧力构件 | 不建议用来命名边界支座，避免与框架/桁架杆件混淆 |

例如门式刚架里的一根斜撑是构件，不是支座；梁底的铰支座、滚动支座、固定端才是边界条件。

## 当前系统事实源

当前支座类型的主要事实源是：

- `shared/supports.json`：梁系、桁架、框架支座目录。
- `frontend/src/lib/support-vocabulary.ts`：前端支座文案、自由度状态、弹性约束说明。
- `backend/contracts/json_schemas.py`：ASMS-JSON 中各分析对象允许的支座枚举。

这说明系统不是三套完全散落的支座定义，而是有共享目录和前后端校验。共享目录负责统一词表，各分析对象负责按自身自由度解释支座约束。

## 当前支座目录总表

### 梁系

梁系当前采用梁弯曲模型。节点自由度是：

- `v`：竖向位移。
- `θz` / `rz`：截面转角。

| 类型值 | 中文名 | 约束自由度 | 释放自由度 | 典型含义 |
|---|---|---|---|---|
| `pinned` | 铰支座 | `v` | `θz` | 约束竖向位移，允许梁端转动 |
| `roller` | 滚动支座 | `v` | `θz` | 当前梁系中与铰支座同自由度，用于表达可水平滑移的竖向支承 |
| `fixed` | 固结支座 | `v / θz` | 无 | 固定端，约束竖向位移和转角 |
| `free` | 自由端 | 无 | `v / θz` | 不提供刚性约束，可叠加弹性约束 |

关键点：当前梁系没有 `ux` 水平位移自由度，所以梁系里的铰支座和滚动支座在计算自由度上相同，都是约束 `v`、释放 `θz`。它们的区别主要是工程语义和图例表达。

### 平面桁架

平面桁架节点自由度是：

- `ux`：全局 X 方向水平位移。
- `uy`：全局 Y 方向竖向位移。

| 类型值 | 中文名 | 约束自由度 | 释放自由度 | 典型含义 |
|---|---|---|---|---|
| `pinned` | 铰支座 | `ux / uy` | 无 | 约束两个平动方向 |
| `roller` | 滚动支座 | `uy` | `ux` | 竖向支承，允许水平滑移 |
| `free` | 自由节点 | 无 | `ux / uy` | 普通杆件连接节点或荷载节点 |

关键点：理想平面桁架节点没有转角自由度，所以桁架支座不提供 `rz` 转动约束，也不产生支座弯矩反力。系统兼容旧版 `fixed` 输入时会归并为 `pinned`，但新模型不应主动暴露桁架 `fixed`。

### 平面框架

平面框架节点自由度是：

- `ux`：全局 X 方向水平位移。
- `uy`：全局 Y 方向竖向位移。
- `rz`：绕 Z 轴平面转角。

| 类型值 | 中文名 | 约束自由度 | 释放自由度 | 典型含义 |
|---|---|---|---|---|
| `fixed` | 固结支座 | `ux / uy / rz` | 无 | 柱脚或梁端完全固结边界 |
| `pinned` | 铰支座 | `ux / uy` | `rz` | 约束平动，允许节点转动 |
| `roller` | 滚动支座 | 默认 `uy`；设置角度时约束法向位移 `n` | 默认 `ux / rz`；设置角度时释放切向位移 `t` 和转角 | 滑动支座，可用角度表达斜向支承面 |
| `free` | 自由节点 | 无 | `ux / uy / rz` | 普通自由节点或无边界节点 |

框架还支持节点弹性约束：

- `ux / uy` 平动弹性约束，单位 `kN/m`。
- `rz` 转动弹性约束，单位 `kN·m/rad`。

关键点：框架是三类对象中支座表达最完整的一类，因为它同时包含平动和转动自由度。

## 同名支座在三类对象中的差异

| 支座名称 | 梁系含义 | 桁架含义 | 框架含义 | 是否完全等价 |
|---|---|---|---|---|
| 铰支座 | 约束 `v`，释放 `θz`；梁系不含 `ux` | 约束 `ux / uy`；桁架无 `rz` | 约束 `ux / uy`，释放 `rz` | 不完全等价 |
| 滚动支座 | 约束 `v`，释放 `θz`；与梁系铰支座同自由度 | 约束 `uy`，释放 `ux` | 默认约束 `uy`，释放 `ux / rz`；可改为斜向法向约束 | 不完全等价 |
| 固结支座 | 约束 `v / θz` | 新模型不支持；旧输入归并为铰支座 | 约束 `ux / uy / rz` | 不等价 |
| 自由端/自由节点 | 释放 `v / θz` | 释放 `ux / uy` | 释放 `ux / uy / rz` | 只在“无边界约束”这个语义上相似 |

这张表是理解系统支座能力的核心：**支座名称可以共享，约束自由度不能脱离分析对象单独解释。**

## 图例建议

下面是概念图例，不是最终 UI 规范。真实 SVG 可以有颜色、比例、选中态和标注差异，但语义应保持一致。

| 图例 | 常见画法 | 梁系 | 桁架 | 框架 |
|---|---|---|---|---|
| 铰支座 | 三角形支承 `△` | 可用；约束 `v` | 可用；约束 `ux / uy` | 可用；约束 `ux / uy` |
| 滚动支座 | 三角形 + 滚轮 `△ ○ ○` | 可用；约束 `v`，与铰支座同自由度 | 可用；约束 `uy` | 可用；默认约束 `uy`，可按角度旋转为法向约束 |
| 固结支座 | 固定块、墙、斜线剖面 `▌` | 可用；约束 `v / θz` | 不建议暴露 | 可用；约束 `ux / uy / rz` |
| 自由端/自由节点 | 无支座符号，或虚线圆点提示 | 可用 | 可用 | 可用 |

### 铰支座

概念草图：

```text
     │ 构件或梁
─────●─────
     △
```

工程含义：

- 允许转动。
- 对框架和桁架，通常约束节点两个平动方向。
- 对当前梁系，因为没有水平自由度，只体现为约束竖向位移、释放转角。

界面提示建议：

```text
铰支座：梁系约束 v；框架/桁架约束 ux、uy。是否存在转角约束取决于分析对象自由度。
```

### 滚动支座

概念草图：

```text
     │ 构件或梁
─────●─────
     △
    ○ ○
```

工程含义：

- 约束一个法向位移。
- 允许沿支承面切向滑动。
- 常与铰支座配合，既保证结构整体稳定，又避免多余水平约束。

对象差异：

- 梁系：当前只约束 `v`，释放 `θz`；与梁系铰支座同自由度。
- 框架：默认约束 `uy`，释放 `ux / rz`；带 `supportAngleDeg` 时约束法向位移 `n`。
- 桁架：约束 `uy`，释放 `ux`。

界面提示建议：

```text
滚动支座：约束支承面法向位移，释放切向位移；梁系当前等效为竖向支承。
```

### 固结支座

概念草图：

```text
墙或固定端
▌────●──── 构件或梁
```

工程含义：

- 既阻止平动，也阻止转动。
- 会产生力反力，也可能产生弯矩反力。
- 常用于悬臂梁固定端、框架柱脚固结、固定边界。

对象差异：

- 梁系：约束 `v / θz`，可产生竖向反力和固定端弯矩。
- 框架：约束 `ux / uy / rz`，可产生水平反力、竖向反力和弯矩反力。
- 桁架：理想桁架无转角自由度，不应把“固结支座”作为新模型支座类型。

界面提示建议：

```text
固结支座：约束可用的平动和转动自由度；桁架模型无转动自由度，不提供固结支座。
```

### 自由端 / 自由节点

概念草图：

```text
────●
    无支座符号
```

工程含义：

- 不提供刚性边界约束。
- 可以是构件端部、普通连接节点或荷载作用节点。
- 框架和梁系可以叠加弹性约束，形成有限刚度边界。

界面提示建议：

```text
自由节点：不提供刚性支座约束；如果设置弹簧，应显示为弹性边界节点。
```

## 支座反力口径

支座反力是支座为了满足约束而产生的反作用。不同支座能产生哪些反力，取决于它约束了哪些自由度。

| 分析对象 | 支座类型 | 可能出现的反力分量 |
|---|---|---|
| 梁系 | 铰支座 / 滚动支座 | 竖向反力 |
| 梁系 | 固结支座 | 竖向反力、弯矩反力 |
| 桁架 | 铰支座 | 水平反力、竖向反力 |
| 桁架 | 滚动支座 | 竖向反力 |
| 框架 | 铰支座 | 水平反力、竖向反力 |
| 框架 | 滚动支座 | 默认竖向反力；斜向支座为法向反力并可投影到全局 X/Y |
| 框架 | 固结支座 | 水平反力、竖向反力、弯矩反力 |

如果某个分析对象没有对应自由度，就不应输出对应反力。例如桁架节点没有 `rz`，所以不应输出“支座弯矩反力”作为桁架主结果。

## 这是不是系统问题

分开看。

### 不是问题的部分

以下属于正常设计：

- 梁系、桁架、框架支座枚举不完全相同。
- 梁系铰支座和滚动支座在当前计算自由度上相同。
- 桁架不支持固结支座。
- 框架支持弹性约束和滚动支座角度，而桁架不支持。

这些差异来自力学模型，而不是实现随意。

### 需要继续补强的部分

以下是产品说明和一致性问题：

1. 梁系铰支座和滚动支座图例不同，但计算自由度相同，必须在界面或说明中写明“梁系中同自由度”。
2. 桁架旧版 `fixed` 输入归并为 `pinned`，应只作为兼容逻辑存在，新建模型不应暴露。
3. 框架滚动支座带角度时，必须说明角度是“法向约束方向”，不是构件方向，也不是荷载方向。
4. 图例和报告应显示“约束自由度”，不能只显示图标。
5. “支撑”一词应避免用于边界支座，防止和斜撑、支撑构件混淆。

## 业界约定俗成的理解

结构力学和工程软件里，支座通常按“自由度约束”理解：

- 简支梁常用一个铰支座和一个滚动支座表示整体稳定且允许温度伸缩；在只考虑竖向弯曲的梁模型里，两者都表现为竖向支承。
- 平面桁架节点按铰接处理，杆件只传递轴力，因此支座只约束 `ux / uy` 这两个平动自由度。
- 平面框架构件能传递弯矩，节点有转角自由度，所以固结、铰接、弹性转动边界会显著影响弯矩和侧移。

所以，同名支座跨对象不完全等价，是业界正常现象。真正不能接受的是系统不说明自由度差异，让用户误解边界条件。

## 建议的呈现规则

### 1. 保留共享目录，但按对象展示

继续以 `shared/supports.json` 作为事实源，但界面不应只显示 `pinned / roller / fixed / free`。建议统一展示：

```text
支座中文名 · 约束自由度 / 释放自由度
```

例如：

- 梁系：`滚动支座 · 约束 v，释放 θz；梁系中与铰支座同自由度`
- 桁架：`滚动支座 · 约束 uy，释放 ux`
- 框架：`滚动支座 · 约束法向位移（90°），释放切向位移、rz`

### 2. 支座图例按对象生成

报告和图形区可以共用视觉风格，但图例表必须按对象生成：

- 梁系图例：铰支座、滚动支座、固结支座、自由端、弹性约束。
- 桁架图例：铰支座、滚动支座、自由节点。
- 框架图例：固结支座、铰支座、滚动支座、斜向滚动支座、弹性边界节点、自由节点。

### 3. 报告中加入支座约束说明

每份计算书建议加入一小节：

```text
支座约束说明
- 本模型自由度：...
- 各支座类型对应约束：...
- 弹性约束：...
- 不适用边界：...
```

这对教学用户、产品经理和非土木背景开发者都很重要，也能减少“为什么同样是滚动支座结果不一样”的误解。

### 4. 文本模型和 API 保持收敛

建议长期保持：

- 梁系允许：`pinned / roller / fixed / free`
- 框架允许：`free / pinned / roller / fixed`
- 桁架允许：`free / pinned / roller`

桁架 `fixed` 只保留在兼容解析层，不进入新 schema 枚举、不进入新 UI 选项。

## 实现与文档复核清单

后续支座相关改动可以按这个清单检查：

1. 当前分析对象有哪些自由度？
2. 支座类型到底约束了哪些自由度？
3. 释放了哪些自由度？
4. 是否会产生对应的支座反力或弯矩反力？
5. 图例是否与自由度约束一致？
6. 同名支座在其他对象中是否有不同含义？
7. 文本模型、API schema、前端选项、报告图例是否一致？
8. 桁架是否误引入转动约束或弯矩反力？
9. 梁系是否误导用户认为滚动支座和铰支座在当前模型中计算不同？
10. 框架滚动支座角度是否明确为法向约束方向？

## 后续建议

短期可以做三件事：

1. 将本页作为支座类型和图例说明的公开入口，不再维护本地副本。
2. 在计算书导出中加入支座约束说明表。
3. 在前端支座选择器中稳定显示“约束自由度摘要”，尤其是梁系滚动支座和桁架不支持固结支座这两处。

长期如果接入三维引擎，支座目录应升级为 capability 驱动：

```text
analysisType
  -> availableDofs
  -> supportTypes
  -> constraintDofs
  -> springDofs
  -> legend
  -> reportExplanation
```

这样二维和三维可以共享“支座是自由度约束”的抽象，但不强行共享完全相同的支座枚举。
