# 中英双语与全球开源定位方案

> 本文档记录 ArchSight Solver 在 v1.2 发布后启动中英双语与全球开源定位升级的判断、范围和实施顺序。它是后续规划备忘，不代表 v1.2 发布前需要追加国际化改造。

## 结论

ArchSight Solver 应支持中英双语，但不应在 v1.2 发布收尾阶段临时追加。推荐在 v1.2 发布后分阶段推进：

1. 先做开源入口双语：README、快速开始、能力边界、API 概览、贡献指南和发布记录。
2. 再做前端工作台 UI 双语：导航、参数面板、模板库、结果指标、错误提示和公开案例入口。
3. 最后做工程交付内容双语：WORD / XLSX 计算书、benchmark 元数据、公开验证报告和导出文件名。

项目对外定位不应是“免费替代商业软件”，而应是：

> 开源、透明、可验证、可复核的结构力学求解器工作台。

对应英文定位建议：

> ArchSight Solver is an open, verifiable structural mechanics workbench for beams, plane frames, and plane trusses.

## 背景判断

结构工程软件的采用具有较高信任成本。用户通常会依据验证资料、参考来源、误差边界、教学与工程案例、团队协作流程以及交付责任边界，判断一个工具是否适合进入学习、复核或工程辅助流程。

商业工程软件的优势通常来自长期案例积累、行业工作流适配、技术支持体系和明确的责任边界。开源结构力学工作台如果缺少验证集、参考来源、误差报告和适用范围说明，即使功能可用，也容易被理解为演示型工具。因此，公开文档不应围绕用户群体或软件获取方式做价值判断，而应把比较口径放在可验证性、透明性、可复现性和适用边界上。

因此，ArchSight Solver 的开源叙事应避开“免费替代商业软件”，优先强调：

- 公开 benchmark、参考来源和容许误差。
- 可复现的 REST API、CLI、MCP tools 和 ASMS-JSON 协议。
- 可审查的结构图、荷载图、内力图、挠度曲线和计算书。
- 适合教学、复核、轻量分析和 Agent 工程流，不替代规范设计、注册工程师签审或商业设计平台中的完整责任链。

英文入口的意义在于降低国际结构工程、工程教育和开源贡献者的进入门槛，并让项目更容易按可验证、可复现、可扩展的开源标准被评价。

## 必要性与复杂度

中英双语对全球开源定位是必要项，但它不是简单翻译任务，而是版本级工程治理任务。

必要性：

- GitHub 访问者通常先通过 README、Quickstart、API 示例和贡献指南判断项目是否值得试用。
- 结构力学、benchmark、CLI/API、ASMS-JSON 和教学算例天然具备跨语言传播价值。
- 英文入口能让国际用户更容易提交 issue、复核 benchmark、接入 API 或补充示例。
- 国际开源可见度提升后，也有助于项目以统一的工程证据、贡献流程和能力边界展示可信度。

复杂度：

- 当前项目规则要求技术文档、审查报告和 UI 文本使用专业简体中文，正式双语前需调整 [AGENTS.md](../AGENTS.md) 的语言规则。
- 前端 UI、模板库、结果指标、敏感性分析选项和默认工程名存在大量中文硬编码。
- 后端 DOCX / XLSX 导出、benchmark 标题、错误信息、公开验证报告和 MCP 描述也包含中文专业文案。
- 工程术语必须保持结构工程英语口径，例如 deflection、displacement、support reaction、bending moment、shear force、member axial force、local y deflection 等，不应被泛化成营销式表达。

## 分阶段路线

### 阶段 1：开源入口双语

目标：让国际用户能理解项目定位、启动方式、能力边界和贡献路径。

范围：

- 新增或拆分英文入口文档：`README.md`、`docs/quickstart.md`、`docs/capabilities.md`、`docs/api-reference.md`、`CONTRIBUTING.md`。
- 保留中文版本，例如 `README.zh-CN.md`、`docs/zh-CN/` 或等效目录结构。
- `CHANGELOG.md` 与中文发布记录建立明确同步关系。
- 在文档顶部提供中英文互链，避免用户迷路。

验收：

- 英文用户能完成安装、启动、一次求解、查看 benchmark 和理解贡献方式。
- 中文文档不丢失专业语境。
- 不改变 v1.2 已发布能力和版本号。

### 阶段 2：前端工作台 UI 双语

目标：让 Web 工作台可以在 `zh-CN` 和 `en-US` 之间切换。

范围：

- 引入轻量 i18n 结构或项目内文案字典；不为国际化单独引入重依赖，除非已有明确维护收益。
- 抽离导航、按钮、页签、参数面板、模板库、结果摘要、图表标题、错误提示和公开案例入口文案。
- 语言选择应保存为用户偏好；默认语言可按浏览器语言或项目设置确定，但必须允许用户手动切换。
- 视觉回归应覆盖至少一个中文桌面场景、一个英文桌面场景和一个移动端关键场景。

验收：

- 三个分析模块的“模板 / 基本 / 对象 / 文本 / 表格”入口在中英文下结构一致。
- 英文专业术语不牺牲结构工程准确性。
- 长英文标签不导致按钮、卡片、图表标题或面板布局溢出。

### 阶段 3：计算书、benchmark 与导出内容双语

目标：让工程交付材料和公开验证材料支持语言选择。

范围：

- DOCX / XLSX 导出支持语言参数，例如随 report options 或项目设置传递。
- benchmark case 的 `title`、`purpose`、`reference`、`method`、`checkedMetrics` 支持双语字段或可派生翻译。
- 公开验证报告、算例目录摘要、导出文件名和图表标题遵循同一语言设置。
- API、CLI、MCP 的错误消息和说明文字需要区分机器稳定字段与人类可读本地化文案。

验收：

- 同一模型在中英文计算书中数值、单位、符号和指标口径完全一致。
- 桁架不引入弯矩主指标，框架和梁系指标口径各自正确。
- benchmark 与计算书引用的 caseId、标准值、容许误差和验证来源不因翻译而改变。

## 同步规则

双语不是一次性翻译，而是发布流程的一部分。后续涉及可见版本、能力边界、公开接口或用户工作流时，必须同步检查：

| 对象 | 同步要求 |
|---|---|
| [README.md](../README.md) | 默认英文或中英文入口必须与当前能力一致，顶部应能跳转到另一语言版本。 |
| [CHANGELOG.md](../CHANGELOG.md) | 发布条目需同步英文和中文，不允许一侧遗漏功能、修复或破坏性变化。 |
| [docs/roadmap.md](roadmap.md) | 路线项涉及全球开源或国际化时，需链接本方案并说明阶段边界。 |
| [docs/quickstart.md](quickstart.md) | 启动、测试、CLI、MCP、端口和环境变量说明需同步。 |
| [docs/capabilities.md](capabilities.md) | 能力边界、非目标和工程责任口径需同步。 |
| [docs/api-reference.md](api-reference.md) | API 字段、错误契约和示例 payload 需同步。 |
| [CONTRIBUTING.md](../CONTRIBUTING.md) | 贡献方式、提交规范、benchmark 投稿方式和语言要求需同步。 |
| [frontend/scripts/sync-release-notes.mjs](../frontend/scripts/sync-release-notes.mjs) | 前端 release notes 生成逻辑需支持语言版本或明确只生成某一语言。 |
| [AGENTS.md](../AGENTS.md) | 当前“强制中文产出”规则需调整为双语治理规则，避免后续实现与项目规则冲突。 |

版本发布检查应继续遵守现有版本一致性要求；如果双语文档引入可见版本号，必须与 `frontend/package.json`、`pyproject.toml`、`uv.lock`、`CHANGELOG.md` 和 README / docs 中的版本展示保持一致。

## 风险与控制

- 术语漂移：建立中英术语表，优先使用结构力学教材、工程软件和国际通用表达。
- 文档分叉：发布前增加中英文文档同步检查项，避免英文落后于中文。
- UI 变形：英文标签更长，必须通过视觉回归检查桌面和移动端布局。
- 维护负担上升：先覆盖开源入口，再根据外部反馈决定是否推进全量 UI 和导出双语。
- 定位失焦：持续强调可验证、可复核、教学与轻量工程分析，不承诺商业设计软件能力。

## 最小验证

阶段 1 完成后，用 4 到 6 周观察以下信号：

- 是否出现非中文用户 issue、discussion、fork、star 或 benchmark 投稿。
- 英文 README 是否能独立支撑安装、启动和一次标准算例求解。
- 国际用户是否能理解 ASMS-JSON、benchmark、CLI/API 和导出计算书的价值。
- 中文用户是否能通过双语入口更清楚地理解项目可信度、能力边界和贡献方式。

如果外部反馈有限，保留英文开源入口即可，暂缓全量 UI 与计算书双语；如果反馈明确，再进入阶段 2 和阶段 3。
